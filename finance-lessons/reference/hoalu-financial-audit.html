Hoalu Financial Audit

Reference — what the app does today, what it lacks, what is financially wrong. Read by every lesson; updated as the app and your understanding evolve.

What Hoalu does today

  • Expense & income tracking — full CRUD, multi-currency, attachments, AI quick-entry, receipt OCR.
  • Recurring bills — schedules (daily/weekly/monthly/yearly), occurrence projection, overdue/today/upcoming view, "log payment" creates an expense and marks an occurrence paid.
  • Event budgets — an event has a budget; the app sums FX-converted expenses linked to it and shows spent vs remaining.
  • Dashboard — period cash-flow area chart, expense/income bar chart with outlier capping, three stat cards (income/expense/transactions) with period-over-period %, category breakdown, upcoming bills, recent transactions.
  • Multi-currency display — FX rate table with time windows; cross-rate via USD; frontend and backend conversion paths.
  • Workspaces & members — multi-tenant, owner/admin/member roles, invitation system.

Financially incorrect or questionable design

1. Wallet has no balance at all schema.ts:218

The wallet table has no opening_balance, current_balance, or balance column. The wallet list renders no balance column; use-wallets.ts computes a row count, not a sum. The app literally cannot show how much money is in any account.

2. Credit card is a plain wallet, not a debt schema.ts:225

wallet.type includes "credit-card" but there is no credit_limit, balance_owed, available_credit, statement_balance, minimum_payment, apr, or due_date. A credit card here is indistinguishable from a cash wallet, so credit-card debt is invisible.

3. Cash-flow "balance" starts from zero each period cash-flow-chart.tsx

The dashboard's area chart plots a running cumulative (income − expense) that starts at 0 for the selected period. It is not opening-balance + transactions. Changing the date range shows wildly different "balances" with no carried-forward figure — visually implying an account balance that doesn't exist.

4. Event currency vs frontend budget_currency mismatch likely bug

DB column is currency; the frontend collection schema expects budget_currency and has no currency field. The sync proxy passes raw DB columns. Either events fail validation or the budget currency is undefined — breaking budget display for any event whose currency differs from the workspace default.

5. Single conversion date for a whole month workspaces/repository.ts

convertAndSum converts all transactions in a month using one month-end rate. An expense on the 1st and the 31st use the same rate, ignoring the rate on the actual transaction date. Misstates monthly totals for volatile currencies.

6. FX lookup uses created_at, not transaction date use-expenses.ts, use-incomes.ts

Frontend FX lookup uses the row's insert timestamp. Backdated expenses get the rate from the data-entry moment, not the transaction date — a temporal mismatch that distorts history.

7. Missing FX rate silently zeroes the amount silent data loss

No rate → multiplier 0 (frontend) or row skipped (backend). A USD expense in a VND workspace with no rate loaded contributes nothing to totals, with no warning to the user.

8. Primary-currency heuristic ignores the workspace setting workspaces/repository.ts

Picks the currency of the wallet with the most expense count, so 100 tiny USD expenses beat 50 large VND ones — even though workspace.metadata.currency is already set on creation.

9. Recurring-bill dueDay=0 is accepted but invalid recurring-bills/schema.ts

Schema allows dueDay from 0 to 31. For a monthly bill, 0 produces new Date(y, m, 0) = last day of the previous month — occurrences land in the wrong month. One field serves both weekly (0–6) and monthly (1–31) frequencies, so it cannot validate correctly.

10. "One-time" and "custom" recurring bills generate nothing repository.ts

generateOccurrences has branches only for daily/weekly/monthly/yearly. A "one-time recurring bill" is a contradiction but the schema permits it, producing zero occurrences silently.

11. expense.repeat and income.repeat are decorative schema.ts

Unless linked to a recurring_bill, the repeat selector on every expense/income form does nothing. The PATCH handler explicitly refuses to auto-create a recurring bill. Income has repeat but no recurringBillId at all. The UI implies automation that does not exist.

12. Floating-point cross-rate maths packages/finance

Rates are stored as numeric(18,6) (exact) but in-memory cross-rate uses parseFloat and float multiplication — inviting rounding drift in cross-currency sums.
